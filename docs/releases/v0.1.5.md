# Zyra v0.1.5 — Núcleo Modular de Comandos

## Visão geral

A versão `0.1.5` marca a transição do Zyra para uma base mais modular na camada de comandos, reduzindo o acoplamento entre o núcleo de conexão e a lógica executada pelos comandos.

O foco desta release foi transformar o processamento de comandos em uma estrutura mais estável, previsível e preparada para crescimento, mantendo o core responsável pela orquestração e abstraindo detalhes do Baileys para um único ponto de manutenção.

Comparação:

- Base anterior: `v0.1.4`
- Nova versão: `v0.1.5`

## Destaques

### Arquitetura modular de comandos

Foi introduzido um runtime interno dedicado para comandos, separado da pasta `src/commands`, que continua sendo responsável apenas pelos módulos funcionais de cada comando.

Com isso, o fluxo passou a seguir um desenho mais claro:

- `src/commands/`: comandos independentes
- `src/core/command-runtime/context.ts`: contexto normalizado entregue aos comandos
- `src/core/command-runtime/processor.ts`: processador central de comandos
- `src/core/command-runtime/admin.ts`: ações administrativas reutilizáveis

Essa separação reduz a chance de mudanças no Baileys afetarem diretamente os arquivos de comando.

### Contexto fechado e desacoplado

Os comandos deixaram de receber `socket` e `message` brutos. Agora recebem um `ctx` controlado pelo core, com:

- dados normalizados do evento
- helpers de resposta
- helpers de reação
- funções administrativas centralizadas

Esse modelo torna os comandos mais simples de manter e menos dependentes dos detalhes internos da conexão.

### Camada administrativa centralizada

Foi criada uma camada reutilizável para operações de grupo, incluindo:

- `isAdmin`
- `kick`
- `ban`
- `promote`
- `demote`

O objetivo é garantir que ajustes futuros de compatibilidade ou correções de bugs sejam feitos em um ponto único do core, sem necessidade de editar todos os comandos que dependem dessas ações.

### Router mais simples

O roteador agora apenas encaminha mensagens recebidas para o processador do core, deixando a responsabilidade de construir contexto, localizar comandos, executar o fluxo e registrar falhas dentro do runtime de comandos.

### Cobertura de testes ampliada

Foram adicionados testes específicos para:

- contexto de comando
- processador de comandos
- integração do roteador com o processador

Com isso, o fluxo modular recém-introduzido deixa de depender apenas da compilação e passa a ter cobertura prática automatizada.

## Melhorias adicionais incluídas no ciclo

Além da refatoração do runtime de comandos, o intervalo entre `v0.1.4` e `v0.1.5` também consolidou:

- documentação dedicada da arquitetura modular de comandos
- ajustes e expansão da documentação principal
- melhorias na configuração de lint
- inclusão de automações com workflows de testes e análise
- consultas SQL adicionais para estatísticas, diagnósticos e análise operacional
- atualização de dependências e sincronização da versão do pacote

## Arquivos e áreas impactadas

Principais áreas afetadas nesta release:

- `src/core/command-runtime/`
- `src/router/`
- `src/commands/`
- `tests/`
- `docs/`
- `package.json`
- `package-lock.json`

## Validação da release

Antes da publicação da `v0.1.5`, a base foi validada com:

- `npm run lint`
- `npm test`
- `npm run build`

Resultado:

- lint sem erros
- `25` testes passando
- build concluído com sucesso

## Impacto esperado

Esta release prepara o Zyra para os próximos passos do sistema de comandos, incluindo:

- middlewares
- permissões por comando
- expansão segura de helpers do core
- evolução futura para carregamento dinâmico de comandos e plugins

## Observações

Esta versão prioriza arquitetura, manutenção e base de crescimento. Embora algumas mudanças sejam internas, elas têm impacto direto na velocidade e segurança das próximas evoluções do projeto.
